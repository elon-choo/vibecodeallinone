# Code Reviewer - 비개발자를 위한 완전 가이드

> **문서 버전**: 1.0.0
> **대상 독자**: 비개발자, 스킬 구조를 이해하고 싶은 사람
> **목표**: 코드 리뷰 스킬이 왜 필요한지, 어떻게 작동하는지 완전히 이해하기

---

## 1. 왜 Code Reviewer가 필요한가?

### 비유로 이해하기: "편집자"

**개발자 = 작가**
소설을 쓰는 데 집중합니다.

**Code Reviewer = 편집자**
- "이 문장이 너무 길어요" (복잡도 분석)
- "앞에서 같은 내용을 썼는데요?" (중복 코드 탐지)
- "독자가 이해하기 어려워요" (가독성 점수)
- "전체적으로 B+ 입니다" (통합 점수)

### 실제 문제: 셀프 체크의 한계

```
[ 셀프 체크가 안 되는 이유 ]

개발자가 직접 작성한 코드를 검토할 때:
1. "내가 쓴 거니까 당연히 이해됨" → 다른 사람은 못 알아봄
2. "이게 최선이야" → 더 좋은 방법이 있음
3. "일단 돌아가니까 OK" → 나중에 버그 폭탄
4. "피곤해서 대충..." → 품질 저하

결과:
- 다른 개발자: "이 코드 뭐야?"
- 6개월 후 본인: "내가 이걸 썼다고?"
```

**Code Reviewer는 "객관적인 기준"으로 코드를 평가합니다.**

## 2. 스킬의 핵심 역할

### 2.1 통합 점수 산출

```
┌─────────────────────────────────────────────────────────────┐
│               Code Reviewer의 점수 체계                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌───────────────┐      가중치                             │
│   │ 코드 품질     │ ───→ 25%   (clean-code-mastery)        │
│   ├───────────────┤                                         │
│   │ 보안 검사     │ ───→ 25%   (security-shield)           │
│   ├───────────────┤                                         │
│   │ 테스트 커버리지│ ───→ 20%   (tdd-guardian)              │
│   ├───────────────┤                                         │
│   │ 문서화        │ ───→ 15%   (주석, README)              │
│   ├───────────────┤                                         │
│   │ 네이밍/스타일 │ ───→ 15%   (일관성)                    │
│   └───────────────┘                                         │
│            │                                                │
│            ▼                                                │
│   ┌───────────────┐                                         │
│   │  총점: 85점   │ → 등급: B (마이너 수정 후 머지)        │
│   └───────────────┘                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 Quality Gate (품질 관문)

**Quality Gate란?**
코드가 통과해야 하는 최소 기준점

```yaml
Blocking_Gates:  # 이것 못 통과하면 머지 불가
  - "TypeScript Strict 에러 없음"
  - "보안 취약점 없음"
  - "하드코딩 시크릿 없음"

Warning_Gates:  # 권고사항, 통과 못해도 머지 가능
  - "복잡도 ≤ 10"
  - "파일 길이 ≤ 300줄"
  - "테스트 커버리지 ≥ 80%"
```

**비유: 공항 보안 검색**
```
Blocking = 금속 탐지기
  → 금속 반응 있으면 탑승 불가

Warning = X-ray 검색
  → 수상한 물건 있으면 추가 검사, 통과는 가능
```

### 2.3 Code Smell 탐지

**Code Smell이란?**
"냄새 나는 코드" - 당장 에러는 안 나지만 문제를 일으킬 코드

```yaml
대표적인_Code_Smells:

  Long_Method:
    냄새: "함수가 50줄이 넘음"
    문제: "이해하기 어렵고 테스트하기 힘듦"
    해결: "작은 함수로 분리"

  Large_Class:
    냄새: "클래스가 500줄이 넘음"
    문제: "책임이 너무 많음 (SRP 위반)"
    해결: "역할별로 클래스 분리"

  Duplicate_Code:
    냄새: "같은 코드가 여러 곳에 있음"
    문제: "수정 시 여러 곳 고쳐야 함"
    해결: "공통 함수로 추출"

  Dead_Code:
    냄새: "실행되지 않는 코드가 있음"
    문제: "코드베이스 복잡도 증가"
    해결: "삭제"

  Feature_Envy:
    냄새: "다른 클래스 데이터를 더 많이 사용"
    문제: "잘못된 위치에 있는 로직"
    해결: "해당 클래스로 메서드 이동"

  God_Class:
    냄새: "모든 걸 다 하는 클래스"
    문제: "수정 시 전체 시스템에 영향"
    해결: "책임 분리"
```

### 2.4 복잡도 분석

**복잡도란?**
코드가 얼마나 복잡한지 수치화한 것

```
[ Cyclomatic Complexity (순환 복잡도) ]

분기점(if, for, while, switch 등) 개수를 세어 복잡도 측정

예시:
┌─────────────────────────────────────┐
│ function process(x) {               │
│   if (x > 0) {           // +1      │
│     if (x > 10) {        // +1      │
│       return 'big';                 │
│     } else {                        │
│       return 'medium';              │
│     }                               │
│   } else {                          │
│     return 'negative';              │
│   }                                 │
│ }                                   │
│                                     │
│ 복잡도 = 1(기본) + 2(분기) = 3     │
└─────────────────────────────────────┘

복잡도 기준:
  1-5:   ✅ 단순, 이해하기 쉬움
  6-10:  ⚠️ 보통, 주의 필요
  11-20: ❌ 복잡, 리팩토링 권장
  21+:   🚨 매우 복잡, 분리 필수
```

**Cognitive Complexity (인지 복잡도)**
```
사람이 코드를 읽을 때 얼마나 힘든지 측정

추가 패널티:
  - 중첩 깊이 (if 안에 if 안에 if...) +3점씩
  - break/continue 사용 +1점
  - 재귀 호출 +1점

인지 복잡도 기준:
  0-8:   ✅ 쉬움
  9-15:  ⚠️ 보통
  16+:   ❌ 어려움, 단순화 필요
```

## 3. 리뷰 프로세스

### 3.1 Self-Review Process

```
┌─────────────────────────────────────────────────────────────┐
│                 코드 리뷰 자동 프로세스                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   1. 코드 생성/수정 완료                                     │
│          ↓                                                  │
│   2. Quick Check (5초)                                      │
│      - 문법 오류                                            │
│      - 타입 에러                                            │
│      - 네이밍 규칙                                          │
│          ↓                                                  │
│   3. Quality Gate Check (10초)                              │
│      - 복잡도 측정                                          │
│      - 중복 코드 탐지                                       │
│      - 의존성 방향 검사                                     │
│          ↓                                                  │
│   4. Security Check (10초)                                  │
│      - 취약점 스캔                                          │
│      - 시크릿 탐지                                          │
│          ↓                                                  │
│   5. Test Coverage Check (5초)                              │
│      - 커버리지 계산                                        │
│      - 누락된 테스트 케이스                                 │
│          ↓                                                  │
│   6. 점수 산출 및 피드백                                     │
│      - 70점 미만: ❌ 수정 필요                              │
│      - 70-85점: ⚠️ 권고 사항 제시                           │
│      - 85점 이상: ✅ 승인                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 Review Levels

```yaml
Quick_Review:  # 5초
  대상: "단일 함수"
  검사항목:
    - 네이밍 규칙
    - 함수 길이
    - 기본 타입 체크

Standard_Review:  # 10초
  대상: "파일 레벨"
  검사항목:
    - Quick 검사 +
    - 복잡도 분석
    - Code Smell
    - 보안 기본

Deep_Review:  # 30초
  대상: "멀티 파일"
  검사항목:
    - Standard 검사 +
    - 파일 간 의존성
    - 아키텍처 패턴
    - 전체 테스트 커버리지
```

## 4. 등급 시스템

### 4.1 점수 계산 방식

```
총점 = (코드품질 × 0.25) + (보안 × 0.25) + (테스트 × 0.20)
     + (문서화 × 0.15) + (네이밍 × 0.15)

예시:
  코드 품질: 90점 × 0.25 = 22.5
  보안:     85점 × 0.25 = 21.25
  테스트:   80점 × 0.20 = 16
  문서화:   70점 × 0.15 = 10.5
  네이밍:   95점 × 0.15 = 14.25
  ──────────────────────────
  총점:     84.5점 → B등급
```

### 4.2 등급별 기준

```
┌─────────────────────────────────────────────────────────────┐
│   등급   │  점수범위  │       의미       │      액션         │
├─────────────────────────────────────────────────────────────┤
│    A     │  90-100   │  우수           │  즉시 머지 가능    │
│    B     │  80-89    │  양호           │  마이너 수정 후    │
│    C     │  70-79    │  보통           │  조건부 승인       │
│    D     │  60-69    │  미흡           │  대폭 수정 필요    │
│    F     │  0-59     │  불합격         │  재작성 필요       │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 등급별 피드백 예시

```markdown
## A등급 (92점)
✅ 코드 품질이 매우 우수합니다.
- SOLID 원칙 준수
- 테스트 커버리지 95%
- 보안 취약점 없음
→ 즉시 머지해도 됩니다!

---

## B등급 (84점)
⚠️ 전반적으로 좋으나 몇 가지 개선점이 있습니다.
- [권고] getUserData() 함수 복잡도가 12입니다 (권장: 10 이하)
- [권고] 테스트 커버리지 78% (권장: 80%)
→ 수정 후 머지하세요.

---

## C등급 (73점)
⚠️ 조건부 승인. 다음 사항을 확인하세요:
- [필수] processOrder() 함수 80줄 → 분리 필요
- [필수] 중복 코드 3곳 발견
- [권고] 문서화 부족
→ 필수 사항 수정 후 다시 리뷰 요청하세요.

---

## F등급 (45점)
❌ 재작성이 필요합니다.
- [심각] SQL Injection 취약점 발견
- [심각] 하드코딩된 비밀번호
- [필수] 테스트 없음
- [필수] 400줄짜리 God Class
→ 보안 취약점부터 해결하세요!
```

## 5. 스킬 작동 방식

### 5.1 자동 발동 조건

```yaml
트리거_키워드:
  한국어:
    - "리뷰해", "검토해", "봐줘"
    - "피드백", "확인해", "체크해"
    - "코드 품질", "분석해"
    - "이거 괜찮아?"

  영어:
    - "review", "check", "feedback"
    - "quality", "analyze", "validate"

자동_실행_조건:
  - Write 도구 사용 직후
  - Edit 도구 사용 직후
  - 파일 생성/수정 완료 시
```

### 5.2 선택적 문서 로드

```
[ 로드 순서 ]

1단계: 언어 감지
  - .ts → TypeScript
  - .py → Python
  - .go → Go
  - .java → Java

2단계: 항상 로드 (공통)
  - core/quality-gates.md (품질 관문)
  - core/code-smells.md (코드 스멜 패턴)
  - core/complexity.md (복잡도 계산)

3단계: 언어별 로드
  - TypeScript → languages/typescript-review.md
  - Python → languages/python-review.md
  - Go → languages/go-review.md

4단계: 상황별 로드
  - 네이밍 검토 → patterns/naming-conventions.md
  - 점수 계산 → patterns/scoring-system.md
```

## 6. 리뷰 체크리스트

### 6.1 Quick Checklist (빠른 검토)

```markdown
## 네이밍
- [ ] 변수명이 의미를 담고 있는가?
- [ ] 함수명이 동사로 시작하는가?
- [ ] 클래스명이 명사인가?
- [ ] 약어 사용을 피했는가?

## 구조
- [ ] 함수가 20줄 이하인가?
- [ ] 들여쓰기가 3단계 이하인가?
- [ ] 파라미터가 3개 이하인가?

## 타입
- [ ] any 타입 사용을 피했는가?
- [ ] null/undefined 처리가 있는가?
```

### 6.2 Standard Checklist (표준 검토)

```markdown
## Quick Checklist +

## 복잡도
- [ ] 순환 복잡도가 10 이하인가?
- [ ] 인지 복잡도가 15 이하인가?

## 중복
- [ ] 3줄 이상 중복 코드가 없는가?
- [ ] 복사-붙여넣기 코드가 없는가?

## 의존성
- [ ] 순환 의존성이 없는가?
- [ ] 의존성 방향이 올바른가?

## 에러 처리
- [ ] 모든 에러 케이스를 처리하는가?
- [ ] 적절한 에러 메시지가 있는가?
```

### 6.3 Deep Checklist (심층 검토)

```markdown
## Standard Checklist +

## 아키텍처
- [ ] 레이어 분리가 명확한가?
- [ ] 비즈니스 로직이 적절한 위치에 있는가?
- [ ] 확장 가능한 구조인가?

## 테스트
- [ ] 단위 테스트가 있는가?
- [ ] 통합 테스트가 있는가?
- [ ] Edge case를 테스트하는가?

## 문서화
- [ ] 복잡한 로직에 주석이 있는가?
- [ ] API 문서가 있는가?
- [ ] README가 최신인가?

## 성능
- [ ] N+1 쿼리 문제가 없는가?
- [ ] 불필요한 반복이 없는가?
- [ ] 메모리 누수 가능성이 없는가?
```

## 8. 구조적 허점 및 개선점

### 8.1 현재 허점

```yaml
허점_1_가중치_고정:
  문제: "가중치가 25/25/20/15/15로 고정됨"
```

## 9. 핵심 용어 사전

| 용어 | 설명 | 비유 |
|------|------|------|
| **Quality Gate** | 통과해야 하는 최소 품질 기준 | 입장 기준 |
| **Code Smell** | 문제를 암시하는 코드 징후 | 상한 음식 냄새 |
| **Cyclomatic Complexity** | 분기점 기반 복잡도 | 미로의 갈림길 수 |
| **Cognitive Complexity** | 이해 난이도 복잡도 | 읽기 난이도 |
| **Dead Code** | 실행 안 되는 코드 | 사용 안 하는 물건 |
| **God Class** | 모든 걸 하는 거대 클래스 | 만능 직원 |
| **Feature Envy** | 다른 클래스 데이터를 더 많이 쓰는 메서드 | 남의 집에 더 자주 있는 사람 |
| **Duplicate Code** | 복사-붙여넣기 코드 | 똑같은 책 여러 권 |
| **Merge** | 코드 합치기 | 원고 출판 |
| **Coverage** | 테스트가 실행하는 코드 비율 | 퀴즈 범위 |
