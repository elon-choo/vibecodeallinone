# 100점 오픈소스 마스터 플랜

## 1. 북극성

이 프로젝트의 최종 목표는 `vibecodeallinone`을 단순한 코드 품질 툴킷이 아니라, 아래 조건을 만족하는 오픈소스 비서 앱과 품질 운영체계로 끌어올리는 것이다.

- 비개발자도 `GPT OAuth`로 쉽게 시작할 수 있어야 한다.
- 메모리 기능이 실제로 동작하고, 저장/검색/수정/삭제/내보내기가 가능해야 한다.
- PC와 모바일 모두에서 쓰기 쉬워야 한다.
- `openclaw보다 더 좋다`는 말을 막연한 주장으로 쓰지 않고, 온보딩, 메모리, UX, 신뢰도, 오픈소스 운영성에서 내부 기준으로 더 높은 점수를 받아야 한다.
- `Ralph Loop`는 자기채점 도구가 아니라 증거 기반 품질 운영체계가 되어야 한다.

## 2. 100점의 정의

| 축 | 배점 | 100점 조건 | 최소 증거 |
|---|---:|---|---|
| 비개발자 온보딩 | 20 | 회원가입, OAuth, 첫 실행까지 막히지 않는다 | 설치 가이드, 데모 영상, 첫 실행 체크리스트 |
| 메모리 품질 | 20 | 장기 메모리가 저장되고 다시 불러와지며 사용자가 통제할 수 있다 | 데이터 모델 문서, CRUD 시연, 삭제/내보내기 경로 |
| PC/모바일 UX | 20 | 같은 핵심 작업을 데스크톱과 모바일에서 무리 없이 수행한다 | 반응형 UX, 모바일 검증, 설치 흐름 |
| 신뢰 가능한 품질 | 20 | 점수와 실제 품질이 분리되지 않는다 | 테스트, E2E, 외부 감사, release evidence |
| 오픈소스 운영성 | 20 | 외부 기여자와 사용자가 재현 가능하게 설치·검증·기여할 수 있다 | README, CONTRIBUTING, SECURITY, 릴리즈 정책 |

## 3. 이번 계획의 출발점

이번 마스터 플랜은 아래 문서들을 기반으로 재구성했다.

- 외부 프롬프트 원본: `/Users/elon/Documents/엘런_모바일연동/vibecodeallinone/Ralph_Loop_v5_Review_Round3.md`
- 외부 피드백 묶음: `/Users/elon/Documents/엘런_모바일연동/vibecodeallinone/AI피드백_v3/`
- 실제 작업 저장소: `/Users/elon/Documents/elon_opensource/claude-code-power-pack/`

리뷰 공통 결론은 아래 5개로 압축된다.

1. `Ralph Loop v5`는 비용과 처리량 면에서는 전진했지만, 점수의 신뢰도는 아직 약하다.
2. 가장 큰 문제는 `score farming` 가능성이다. 특히 Stage 2 self-review와 artifact 무결성이 취약하다.
3. 산식, 정책, 입출력이 흩어져 있어 단일 진실 공급원이 없다.
4. 테스트는 양보다 순서가 중요하다. `server.py`, 검색 파이프라인, `feedback_loop.py`, `ralphloop`부터 덮어야 한다.
5. 오픈소스 경쟁력은 코드만이 아니라 문서, 릴리즈 증거, 외부 감사 체계까지 포함해야 생긴다.

## 4. 전략 원칙

1. 제품 목표가 먼저고 `Ralph Loop`는 수단이다.
2. 비개발자 기준의 사용성 없이는 100점이 아니다.
3. 메모리는 기능이 아니라 신뢰 계약이다. 저장 근거와 사용자 통제권이 있어야 한다.
4. 모바일은 나중 보완 항목이 아니라 초기 설계 조건이다.
5. 점수보다 증거를 우선한다.
6. 모든 세션은 문서로 시작하고 문서로 끝낸다.

## 5. 세션 운영 규칙

### 시작 규칙

각 세션은 아래 문서를 이 순서대로 읽고 시작한다.

1. `docs/session-ops/00_MASTER_PLAN.md`
2. `docs/session-ops/01_SESSION_BOARD.md`
3. 최신 `docs/session-ops/handovers/SESSION_XX_HANDOVER.md`
4. 최신 `docs/session-ops/prompts/SESSION_XX_PROMPT.md`

### 진행 규칙

- 세션 도중 계획, 범위, 순서가 바뀌면 먼저 `01_SESSION_BOARD.md`의 `Plan Change Log`를 수정한다.
- 세션이 새로운 문서를 만든다면, 그 문서 경로를 `01_SESSION_BOARD.md`에 등록한다.
- 코드 변경을 시작하기 전에는 해당 세션의 목표와 종료 조건이 문서에 이미 있어야 한다.

### 종료 규칙

각 세션이 끝나면 반드시 아래 4가지를 남긴다.

1. 작업 결과 반영된 `01_SESSION_BOARD.md`
2. `handovers/SESSION_XX_HANDOVER.md`
3. `prompts/SESSION_YY_PROMPT.md`
4. 필요 시 새 산출물 문서

더 이어갈 세션이 없으면 `prompts/SESSION_YY_PROMPT.md` 대신 루트 `NEXT_SESSION_PROMPT.md`에 stop marker만 남긴다.

## 6. 세션 로드맵

현재 권장 구조는 총 9개 세션이다. 실제 진행 중 세션을 합치거나 나눌 수 있고, 그 경우 `01_SESSION_BOARD.md`를 갱신한다.

| Session | 1차 목적 | 필수 산출물 | 종료 조건 |
|---|---|---|---|
| S00 | 세션 운영 체계 구축 | 마스터 플랜, 세션 보드, 템플릿, S00 handover, S01 prompt | 다음 세션이 문서만 읽고 바로 이어갈 수 있음 |
| S01 | 제품 정의 고정 | Product Brief, Win Rubric, MVP/V1 범위 문서 | 최종 목표와 승부 기준이 문서로 고정됨 |
| S02 | 시스템 아키텍처 확정 | 아키텍처 문서, 워크스트림 분해, 데이터/인증/메모리 설계 | 구현 순서와 리포 구조가 명확함 |
| S03 | `Ralph Loop` 신뢰도 복구 | 품질 운영 모델, 점수 무결성 개선 계획, 구현 착수 | score farming 차단 방향이 코드 수준으로 정의됨 |
| S04 | 백엔드 기반 구축 | OAuth, 메모리 스키마, API 설계/구현 문서 | 인증과 메모리의 뼈대가 동작 가능 상태 |
| S05 | 비서 앱 UI 셸 구축 | 웹/모바일 공통 UX 문서, 핵심 화면 구현 | 비개발자 기준 첫 사용 흐름이 보임 |
| S06 | 메모리 UX와 크로스디바이스 완성 | 메모리 관리 UX, 동기화/세이프가드 문서 | PC/모바일에서 같은 핵심 작업이 이어짐 |
| S07 | 검증·문서·기여 체계 강화 | 테스트, 설치, 릴리즈, 기여자 문서 | 외부 사용자가 재현 가능함 |
| S08 | 외부 감사와 출시 후보 마감 | 외부 리뷰 번들, 릴리즈 evidence, launch checklist | 100점에 가까운 공개 후보 상태 도달 |

## 7. 세션별 초점

### S01에서 반드시 결정할 것

- 누구를 위한 비서 앱인지
- `openclaw보다 더 좋다`를 어떤 내부 평가표로 정의할지
- MVP와 V1을 어디서 자를지
- 앱 표면을 새 패키지로 만들지, 현재 저장소 안에 통합할지

### S02에서 반드시 결정할 것

- 웹 우선인지, PWA 우선인지, 네이티브 래핑 시점은 언제인지
- OAuth 흐름과 메모리 저장소 책임 분리
- 현재 `KG MCP + Ralph Loop`를 제품 품질 엔진으로 어떻게 연결할지

### S03에서 반드시 해결할 것

- Stage 2 self-review override 제거 여부
- artifact metadata, atomic write, manifest, history 도입
- `scoring.py`, `io.py`, `policy.py`, `cli.py` 방향
- 80/90 외부 감사 게이트 설계

## 8. 현재 우선순위

지금 당장 중요한 우선순위는 아래 순서다.

1. 제품 정의를 먼저 잠근다.
2. 그 다음 구현 구조를 잠근다.
3. 그 다음 `Ralph Loop`를 신뢰 가능한 품질 운영체계로 고친다.
4. 그 이후에 OAuth, 메모리, UX 구현에 들어간다.

이 순서를 바꾸면 기능은 빨리 늘어도 방향이 흔들릴 가능성이 높다.
