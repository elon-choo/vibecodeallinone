# S01 Win Rubric

## 1. 목적

`openclaw보다 더 좋다`는 표현을 막연한 슬로건으로 쓰지 않고, 동일한 시나리오와 동일한 증거 기준으로 비교하기 위한 내부 평가표다.

이 문서가 없으면 앞으로는 "느낌상 더 낫다"는 말만 남고, 있으면 언제 어떤 조건에서 이겼는지 기록할 수 있다.

## 2. 기본 규칙

1. 비교는 반드시 같은 날짜, 같은 평가자, 같은 시나리오 세트로 진행한다.
2. 비교 대상이 `openclaw`든 다른 제품이든, 원본 메모와 화면 기록을 남긴다.
3. 점수는 데모 감상평이 아니라 실제 실행 증거를 기준으로 준다.
4. `Ralph Loop` 내부 점수만으로는 승리 선언을 할 수 없다.
5. 한 축이라도 증거가 없으면 해당 축은 최고점으로 올리지 않는다.

## 3. 비교 시나리오 세트

| 시나리오 | 무엇을 본다 | 최소 증거 |
|---|---|---|
| 첫 실행 | 로그인부터 첫 답변까지 막힘이 없는지 | 시간 기록, 화면 녹화, 실패 지점 메모 |
| 메모리 저장 | 중요한 정보를 저장하고 다시 꺼낼 수 있는지 | 저장/조회 화면, 데이터 예시 |
| 메모리 통제 | 수정, 삭제, 내보내기가 실제로 되는지 | 각 액션 결과 캡처 |
| 크로스디바이스 | PC에서 시작한 작업을 모바일에서 이어갈 수 있는지 | 두 기기 흐름 기록 |
| 오픈소스 재현 | 외부 사용자가 설치, 검증, 기여 흐름을 따라갈 수 있는지 | 문서 체크리스트, 실행 로그 |

## 4. 100점 구성

마스터 플랜의 5축을 그대로 사용한다.

| 축 | 배점 | 질문 |
|---|---:|---|
| 비개발자 온보딩 | 20 | 로그인과 첫 가치 경험이 쉬운가 |
| 메모리 품질 | 20 | 기억이 저장되고 다시 쓰이며 사용자가 통제 가능한가 |
| PC/모바일 UX | 20 | 데스크톱과 모바일에서 같은 핵심 작업이 이어지는가 |
| 신뢰 가능한 품질 | 20 | 점수와 실제 품질이 분리되지 않는가 |
| 오픈소스 운영성 | 20 | 외부 사용자가 설치, 검증, 기여를 재현할 수 있는가 |

## 5. 공통 채점 기준

| 점수대 | 의미 |
|---|---|
| 0 | 핵심 흐름이 깨졌거나 증거가 없다 |
| 5 | 데모는 되지만 실제 사용에는 막힘이 크다 |
| 10 | 제한적으로 동작하지만 수작업/우회가 많다 |
| 15 | 일반 사용이 가능하고 큰 결함은 적다 |
| 20 | 비개발자 기준에서도 신뢰 가능하고 증거가 충분하다 |

## 6. 축별 세부 평가표

### A. 비개발자 온보딩 20점

| 세부 항목 | 점수 |
|---|---:|
| 로그인 진입점이 명확하고 설명이 쉽다 | 5 |
| 첫 실행 설정이 짧고 막힘이 적다 | 5 |
| 첫 답변 또는 첫 성공 경험까지의 시간이 짧다 | 5 |
| 로그인 실패/권한 문제 시 복구 경로가 보인다 | 5 |

최저 조건:

- 비개발자가 문서 없이도 로그인 경로를 찾을 수 있어야 한다.
- 첫 성공 경험이 "설정만 하다 끝나는 상태"여서는 안 된다.

### B. 메모리 품질 20점

| 세부 항목 | 점수 |
|---|---:|
| 메모리를 명시적으로 저장할 수 있다 | 5 |
| 저장된 메모리를 다시 찾고 활용할 수 있다 | 5 |
| 수정, 삭제, 내보내기 경로가 있다 | 5 |
| 메모리 출처와 통제권이 사용자에게 보인다 | 5 |

최저 조건:

- CRUD 중 하나라도 빠지면 15점을 넘길 수 없다.
- 삭제와 내보내기가 없으면 "기억은 하지만 통제는 못 하는 제품"으로 본다.

### C. PC/모바일 UX 20점

| 세부 항목 | 점수 |
|---|---:|
| 데스크톱에서 핵심 작업이 자연스럽다 | 5 |
| 모바일에서도 같은 핵심 작업이 가능하다 | 5 |
| 기기 전환 후 맥락이 유지된다 | 5 |
| 반응형/터치 환경에서 막힘이 적다 | 5 |

최저 조건:

- PC 전용 데모만 가능하면 10점을 넘길 수 없다.
- 이어쓰기 흐름이 수동 복사 수준이면 15점을 넘길 수 없다.

### D. 신뢰 가능한 품질 20점

| 세부 항목 | 점수 |
|---|---:|
| 테스트와 E2E가 실제 핵심 흐름을 덮는다 | 5 |
| 자기채점이 아니라 독립 검증 레이어가 있다 | 5 |
| artifact/manifest/history 등 증거 체계가 있다 | 5 |
| 실패 상태와 한계가 숨겨지지 않는다 | 5 |

최저 조건:

- 자기채점만으로는 10점을 넘길 수 없다.
- release evidence가 없으면 15점을 넘길 수 없다.

### E. 오픈소스 운영성 20점

| 세부 항목 | 점수 |
|---|---:|
| README가 비개발자와 기여자 모두에게 유용하다 | 5 |
| CONTRIBUTING, SECURITY, release 정책이 실질적이다 | 5 |
| 설치와 검증 절차가 재현 가능하다 | 5 |
| 외부 리뷰/감사 근거를 릴리즈에 연결할 수 있다 | 5 |

최저 조건:

- 파일 존재만으로 점수를 주지 않는다.
- 외부 사용자가 재현할 수 없으면 10점을 넘길 수 없다.

## 7. 승리 선언 규칙

`openclaw보다 더 좋다`고 말하려면 아래 조건을 모두 만족해야 한다.

1. 총점이 `85점 이상`이어야 한다.
2. `온보딩`, `메모리`, `PC/모바일 UX`는 각각 `15점 이상`이어야 한다.
3. `신뢰 가능한 품질`, `오픈소스 운영성`은 각각 `16점 이상`이어야 한다.
4. 동일 시나리오 비교에서 상대보다 `총점 10점 이상` 높거나, `5축 중 3축 이상`에서 `2점 이상` 앞서야 한다.
5. 어느 한 축에서도 `3점 이상` 크게 뒤지면 승리 선언을 보류한다.

## 8. 승리 선언 금지 규칙

아래 조건이면 총점이 높아도 외부적으로 "더 좋다"고 말하지 않는다.

1. 메모리 삭제 또는 내보내기 기능이 없다.
2. 모바일 핵심 흐름이 데모 수준이다.
3. 품질 증거가 `Ralph Loop` 자기점수에 과도하게 의존한다.
4. 설치/검증 문서가 외부 사용자 기준으로 재현되지 않는다.
5. 비교 로그와 증거 파일이 남아 있지 않다.

## 9. 운영 메모

이번 세션에서는 평가표만 고정한다. 실제 `openclaw` 비교 실행은 S07~S08의 검증/외부 감사 단계에서 수행한다.

그 전까지 이 문서는 아래 역할을 한다.

1. S02 아키텍처가 무엇을 우선해야 하는지 정하는 기준
2. S04~S06 구현 범위를 자르는 기준
3. S07~S08에서 어떤 증거를 남겨야 하는지 정하는 기준
