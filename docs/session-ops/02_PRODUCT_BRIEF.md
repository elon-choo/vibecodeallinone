# S01 Product Brief

## 1. 한 줄 정의

`vibecodeallinone`의 다음 제품은 `비개발자가 GPT OAuth(OpenAI-first)로 바로 시작해, 기억을 저장·통제하고, PC와 모바일에서 이어서 쓰는 오픈소스 개인 비서 앱`이다.

## 2. 왜 이 제품을 만드는가

현재 저장소는 `AI 스킬 + KG MCP + Ralph Loop` 중심의 품질 툴킷이다. 강점은 분명하지만, 비개발자가 바로 쓰는 제품 표면은 아직 없다.

이번 제품 정의의 목적은 아래 4가지를 하나의 사용자 경험으로 묶는 것이다.

1. 가입과 첫 실행이 쉬워야 한다.
2. 메모리가 실제로 저장되고 다시 불러와져야 한다.
3. 데스크톱과 모바일에서 같은 핵심 작업이 이어져야 한다.
4. 품질 점수는 내부 만족이 아니라 사용자가 신뢰할 수 있는 증거로 연결되어야 한다.

## 3. 목표 사용자

### 1차 사용자

- 개발자가 아닌 1인 창업가, 운영 담당자, 크리에이터, 지식노동자
- 프롬프트를 길게 설계하기보다 "로그인 후 바로 쓰는 AI 비서"를 원하는 사용자
- 대화 내용, 선호, 작업 맥락을 기억해 주길 원하는 사용자

### 2차 사용자

- AI 도구를 여러 개 써 봤지만, 메모리와 재시작 경험 때문에 이탈한 파워 유저
- 오픈소스라서 설치, 검증, 기여 근거를 확인하고 싶은 초기 기여자

### 이번 단계에서 직접 겨냥하지 않는 사용자

- 복잡한 개발자 IDE 자동화가 1순위인 사용자
- 멀티 에이전트 워크플로를 바로 세밀하게 편집하려는 사용자
- 대규모 팀 협업과 관리자 콘솔이 꼭 필요한 엔터프라이즈 구매자

## 4. 핵심 작업(Job To Be Done)

1. "AI를 열자마자 로그인하고 바로 도움을 받겠다."
2. "내 취향, 프로젝트 맥락, 자주 쓰는 정보를 기억하게 하겠다."
3. "나중에 다시 열었을 때 그 기억을 불러오고 수정하거나 지우겠다."
4. "컴퓨터에서 하던 일을 모바일에서 자연스럽게 이어가겠다."
5. "왜 이 답이 나왔는지, 내 데이터가 어떻게 다뤄지는지 확인하겠다."

## 5. 제품 약속

이 제품은 "똑똑한 챗 UI"가 아니라 아래 약속을 지켜야 한다.

- 로그인 후 첫 가치 경험까지의 거리가 짧다.
- 메모리는 저장만 되는 것이 아니라 사용자가 통제할 수 있다.
- 화면이 바뀌어도 같은 사람의 맥락이 유지된다.
- 품질은 자기 점수가 아니라 테스트, 기록, 감사 근거로 설명된다.
- 오픈소스 프로젝트답게 외부 사용자가 설치와 검증을 재현할 수 있다.

## 6. 핵심 사용자 흐름

1. 사용자가 랜딩 또는 앱 첫 화면에서 `GPT OAuth`로 로그인한다.
2. 첫 실행에서 사용 목적, 개인정보 안내, 메모리 동의 범위를 짧게 설정한다.
3. 기본 홈에서 비서와 대화하거나 추천 액션을 선택한다.
4. 사용자는 특정 정보를 메모리로 저장하거나 자동 저장 후보를 승인한다.
5. 이후 세션에서 저장된 메모리를 검색, 수정, 삭제, 내보내기 할 수 있다.
6. 같은 계정으로 모바일 또는 다른 PC에서 이어서 작업한다.
7. 설정 또는 결과 화면에서 메모리 출처, 최근 변경, 품질 근거를 확인한다.

## 7. 제품 원칙

1. `Ralph Loop`는 사용자에게 전면 노출되는 제품이 아니라 뒤에서 품질을 보증하는 엔진이다.
2. 메모리는 "편리한 기능"이 아니라 사용자와의 신뢰 계약이다.
3. 모바일은 축소판이 아니라 초기 설계 기준이다.
4. 비개발자 온보딩은 기술적 우회로가 아니라 기본 경로여야 한다.
5. "더 좋다"는 주장보다 "왜 더 좋은지 증거를 남기는 구조"가 먼저다.

## 8. 범위 결정

### MVP

- `GPT OAuth(OpenAI-first)` 단일 진입점
- 반응형 웹 앱 또는 PWA 우선
- 개인 메모리 CRUD: 저장, 검색, 수정, 삭제, 내보내기
- PC/모바일 공통 핵심 흐름: 로그인, 대화, 메모리 저장, 메모리 관리, 이어쓰기
- 품질 근거의 최소 표면: 최근 변경 이력, 메모리 통제 경로, 릴리즈 검증 링크

### V1

- 다중 모델 연결 여부 검토 및 확장
- 자동 메모리 추천 정확도 개선
- 공유 가능한 메모리/프로젝트 공간
- 설치형 래퍼(데스크톱, 모바일) 필요성 검증 후 추가
- 품질 근거 대시보드 고도화

### Post-V1

- 팀 협업 기능
- 고급 워크플로 자동화
- 외부 서비스 연동과 에이전트 확장 마켓

## 9. 이번 세션에서 고정하는 제품 결정

1. `GPT OAuth`의 제품 범위는 `OpenAI-first`로 시작한다.
2. 메모리는 `클라우드 동기화 우선 + 로컬 캐시 보조` 방향으로 설계한다.
3. 앱 표면은 `웹/PWA 우선`으로 시작하고, 네이티브 래퍼는 뒤로 미룬다.
4. 사용자에게 먼저 보여줄 핵심 기능은 `대화`, `메모리`, `크로스디바이스 이어쓰기`다.
5. 제품 신뢰 표면은 `Ralph Loop` 점수 자체가 아니라 `증거 묶음`이어야 한다.

## 10. 저장소 구조 제안

이번 단계에서는 앱을 별도 저장소로 분리하지 않고, 현재 저장소 안에 새 제품 패키지를 추가하는 방향을 권장한다.

### 권장안

- 같은 저장소 안에서 `품질 엔진`, `KG MCP`, `사용자 앱 표면`을 함께 관리한다.
- S02에서 `apps/assistant-web` 같은 앱 패키지와 공통 코어/서비스 경계를 정의한다.

### 이유

1. 지금 가장 중요한 것은 새 제품과 품질 엔진 사이의 증거 연결이다.
2. 초기에 저장소를 분리하면 README, 릴리즈, 검증 기준이 다시 흩어진다.
3. 아직 제품과 엔진의 인터페이스가 고정되지 않았으므로, 한 저장소 안이 설계 비용이 낮다.

### 나중에 분리해도 되는 조건

- 앱 사용자가 엔진 사용자보다 훨씬 많아졌을 때
- 릴리즈 주기와 배포 책임이 완전히 달라졌을 때
- 보안/운영 정책상 별도 저장소가 명확히 유리할 때

## 11. S02로 넘기는 설계 질문

1. `GPT OAuth(OpenAI-first)`를 어떤 인증 구조로 구현할지
2. 메모리의 시스템 오브 레코드는 어디에 둘지
3. 앱, API, 품질 엔진, MCP 서버 경계를 어떤 패키지 구조로 나눌지
4. 모바일 이어쓰기의 최소 UX를 어떤 데이터 흐름으로 보장할지
5. 품질 증거를 사용자 표면과 릴리즈 표면에 각각 어떻게 노출할지
